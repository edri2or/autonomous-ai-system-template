# ADR-0105: Centralized Secrets Hub for Multi-Project Setup

**Date:** 2026-04-29  
**Status:** ACCEPTED  
**Phase:** 4 — Architecture Decision Records

---

## Problem

Bootstrap automation creates isolated GCP projects for each new autonomous AI system instance. The original design required:

- Each project creates its own GitHub App (redundant)
- Each project stores provider credentials (railway-api-token, cloudflare-api-token) locally
- No shared credential registry across projects
- Secrets are duplicated in multiple projects (DRY violation)

### Example: GitHub App Management

**Current (Problematic):**
```
Template repo —[bootstrap]—> Project A (creates GitHub App #1)
            |—[bootstrap]—> Project B (creates GitHub App #2)
            |—[bootstrap]—> Project C (creates GitHub App #3)
```

**Issue:** Three redundant GitHub Apps, each requiring manual registration and key storage.

---

## Decision

Implement a **centralized secrets hub** — a single GCP project (default: `or-infra-admin-hub`) that holds:

1. **GitHub App credentials** (single App, shared across all projects)
   - `github-app-private-key` (PEM)
   - `github-app-id` (integer)

2. **Provider tokens** (organization-level credentials)
   - `railway-api-token`
   - `cloudflare-api-token`
   - `cloudflare-zone-id`

### Architecture

```
Hub Project (or-infra-admin-hub)
├── github-app-private-key
├── github-app-id
├── railway-api-token
└── cloudflare-api-token

Project A ─[IAM binding]→ reads from Hub
Project B ─[IAM binding]→ reads from Hub
Project C ─[IAM binding]→ reads from Hub
```

### Workflow Changes

**Bootstrap Phase 1 (GitHub App):**
- ✅ OLD: Create new GitHub App, store in new project
- ✅ NEW: Read existing GitHub App from hub (via `--secrets-hub-project` parameter)

**Terraform Phase (IAM Setup):**
- ✅ NEW: Grant new project's SA `secretAccessor` role on hub secrets

**PHASE 4.5 (Provider Credentials):**
- ✅ OLD: Store in new project
- ✅ NEW: Store in hub (pre-bootstrap.sh reads via hub project)

---

## Implementation

### 1. pre-bootstrap.sh

**New Parameter:**
```bash
--secrets-hub-project HUB_PROJECT_ID  (default: or-infra-admin-hub)
```

**Phase 1 (GitHub App):**
```bash
# Read from hub instead of creating new
if gcloud secrets describe "github-app-id" --project="$SECRETS_HUB_PROJECT" &>/dev/null; then
  # Hub exists with credentials, use them
  APP_ID=$(gcloud secrets versions access latest \
    --secret="github-app-id" --project="$SECRETS_HUB_PROJECT")
else
  # Hub doesn't exist or missing secrets, create new
  # (backward compat: create app + store in hub)
fi
```

**Phase 4.5 (Provider Tokens):**
```bash
# Store provider tokens in hub, not new project
gcloud secrets versions add "railway-api-token" \
  --project="$SECRETS_HUB_PROJECT" --data-file=-
```

### 2. Terraform (cross-project-iam.tf)

**New IAM Bindings:**
```hcl
resource "google_secret_manager_secret_iam_member" "github_app_id_from_hub" {
  secret_id = "projects/${var.secrets_hub_project_id}/secrets/github-app-id"
  role      = "roles/secretmanager.secretAccessor"
  member    = "serviceAccount:${google_service_account.terraform.email}"
}
```

Each project's Terraform SA gets read-only access to:
- `github-app-private-key` (always)
- `github-app-id` (always)
- `railway-api-token` (if enable_railway = true)
- `cloudflare-api-token` + `cloudflare-zone-id` (if enable_cloudflare = true)

### 3. GitHub Secrets

**Phase 4 (pre-bootstrap.sh):**
New secret added to each project:
```bash
set_github_secret "$ORG/$NEW_REPO" "GCP_SECRETS_HUB_PROJECT" "or-infra-admin-hub"
```

This allows workflows in each project to access the hub when needed.

### 4. secret.tf (Local Secrets)

**Old Behavior:** Create containers for ALL secrets locally
**New Behavior:** Only create containers for non-hub secrets:
- `railway-project-id`, `railway-environment-id` (project-specific, generated post-deploy)
- `n8n-encryption-key`, `n8n-owner-password`, `n8n-secrets-broker-token` (project-specific, generated by populate-secrets.yml)

## Consequences

### ✅ Positive

1. **Single Source of Truth** — One GitHub App credential, shared safely across projects
2. **Least Privilege** — Each project's SA has explicit IAM bindings (not global access)
3. **Compliance** — Follows Google Cloud best practices for multi-project credential management
4. **Efficiency** — No redundant credential creation
5. **Backward Compat** — Optional parameter; defaults to standard hub project

### ⚠️ Constraints

1. **Hub Project Required** — Assumes `or-infra-admin-hub` exists (or custom project specified)
2. **Cross-Project IAM** — Requires organizational IAM permissions (not available in isolated sandboxes)
3. **Network Isolation** — Hub project must be in same GCP organization as new projects
4. **Bootstrap Order** — Hub must be provisioned BEFORE first project bootstrap

## Evidence

**Industry Standard:**
- Google Cloud official: "Centralized secret management for multi-project setups"
- Terraform module `terraform-google-gsa`: Examples for multi-project secret access
- External Secrets Operator, HashiCorp Vault Sync: Same cross-project pattern

**Prior Art in Organization:**
- project-life-130: Partial cross-project credentials
- bootstrap-infra: Existing hub project infrastructure
